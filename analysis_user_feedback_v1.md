## 2. 用戶角度分析：還缺什麼

### 2.1 用戶已提到的（明確需求）

| 需求 | 重要性 | 備註 |
|------|--------|------|
| 股票搜尋 + 多市場 | 高 | 目前只有 5 個 demo 案例，無法查任意股票 |
| 消息輸入表單 | 高 | 目前只能改 JSON，非技術用戶無法操作 |
| 結構化輸入（選擇題/日曆） | 中 | 降低輸入門檻，確保數據品質 |

### 2.2 用戶沒提到但一定會遇到的

| 缺口 | 為什麼會遇到 | 影響 |
|------|-------------|------|
| **消息時效性** | 消息有保鮮期。今天聽到「某主管要走」，3 天後才想起要輸入，記憶已模糊。時間越晚輸入，「獲得消息日期」越不準 | 影響消息→股價的時間差分析準確性 |
| **消息可信度校準** | 不同來源可信度差異大（董事會成員 vs 茶水間八卦），但用戶填 confidence 時缺乏參考基準 | 影響影響力模型的訓練品質 |
| **回溯驗證** | 輸入消息後，用戶想知道「這條消息到底準不準」，需要回頭看股價是否如預期變動 | 沒有這個功能，用戶就只是在記帳，感受不到系統價值 |
| **消息去重** | 同一件事可能從不同管道聽到多次（A 說主管要走，B 也說），需要識別為同一事件 | 避免重複標記造成分析偏差 |
| **多人協作衝突** | CEO 和天使投資可能同時輸入同一支股票的消息 | 需要合併或標記來源 |
| **歷史消息批量導入** | 天使投資有大量歷史聊天記錄待整理，不可能一條條手動填 | 第一批數據的冷啟動問題 |

## 3. 技術角度分析：三階段路徑

### 階段 1：前端表單（短期，1-2 週可交付）

**做什麼：** 在現有 Demo HTML 上加輸入功能

```
[股票搜尋欄] → 選市場(港/美/A/台) → 輸入代碼 → 自動拉取股價 → 顯示圖表
[消息輸入表單] → 結構化欄位 → 存入 localStorage / 下載 JSON
```

**表單欄位設計：**
| 欄位 | 輸入方式 | 必填 | 備註 |
|------|---------|------|------|
| 關聯股票 | 下拉選擇（已載入的股票） | 是 | 限選已有數據的股票，避免亂填 |
| 事件標題 | 文字輸入（限 30 字） | 是 | |
| 事件描述 | 多行文字 | 是 | |
| 消息取得日期 | 日曆選擇器 | 是 | 預設今天 |
| 推測事件發生日期 | 日曆選擇器 + 「無法推測」checkbox | 否 | 勾選無法推測則留空 |
| 消息類別 | 單選（私人/企業/宏觀/政策/法律） | 是 | |
| 預期影響方向 | 單選（利好/利空/不確定） | 是 | |
| 信心度 | 滑桿 0-100% | 是 | 預設 50% |
| 消息來源層級 | 單選（董事會/高管/中層/業界/網路） | 否 | 輔助可信度校準 |
| 來源連結 | URL 輸入 | 否 | 公開消息填，私人消息留空 |

**存儲方案：**
- 純前端 → localStorage + 匯出 JSON 按鈕
- 不需要後端，CEO 定期匯出 JSON 上傳 GitHub 即可更新
- 缺點：數據在瀏覽器裡，換電腦就沒了

**股票搜尋方案：**
- 即時股價：呼叫 Yahoo Finance API（前端 fetch，有 CORS 問題）
- 務實方案：預載一份港股/美股/A股的代碼清單（JSON），用戶選好後從 Yahoo Finance URL 導向或用 proxy 拉數據
- 更務實：先支援手動上傳 price JSON（用 fetch_prices.py 拉好再放進來）

### 階段 2：輕量後端（中期，需 1-2 個月開發）

**做什麼：** 加後端處理自動化

```
用戶對話紀錄 → 後端解析 → 自動產生事件 JSON → 推送到 GitHub / 雲端
```

**架構：**
```
[微信聊天記錄導出] → [Python 解析腳本] → [LLM 提取事件] → [events.json]
                                                              ↓
[Dashboard 前端] ←── fetch ←── [GitHub Pages / Cloudflare Pages]
```

**關鍵挑戰：**
| 挑戰 | 難度 | 說明 |
|------|------|------|
| 微信聊天記錄導出 | 高 | 微信加密 DB，CEO 負責處理（會議已分工） |
| LLM 事件提取 | 中 | 從非結構化對話中提取「誰說了什麼、關於哪家公司、什麼時候」|
| 股票代碼匹配 | 中 | 「建行」→ 0939.HK，需要一份公司名→代碼的對照表 |
| 自動 vs 人工審核 | 低 | 建議 LLM 提取後人工確認再入庫，避免錯誤數據污染 |

### 階段 3：全自動化（長期願景）

```
[多源自動抓取] → [LLM 分類+影響力評估] → [自動更新 Dashboard] → [推送通知]
```

**輸入源：**
| 來源 | 自動化可行性 | 備註 |
|------|-------------|------|
| 微信聊天記錄 | 中（需定期導出） | CEO 負責導出腳本 |
| 微信公眾號 | 中（需爬蟲或 RSS 代理） | 可用 WeChat RSS 第三方工具 |
| 財經新聞 API | 高 | 新浪財經/東方財富有公開 API |
| 港交所公告 | 高 | HKEX News 有 RSS |
| SEC/EDGAR | 高 | 美股公告 API 成熟 |
| TWSE 公告 | 高 | 台灣證交所 OpenData |
| 巨潮資訊 | 中 | A 股公告，需爬蟲 |

## 5. John 說得對的部分（強調）

> 「手動輸入意味著使用者要改變行為習慣，記得按照格式輸入這個系統，否則資料就不全」

這是核心問題。**任何需要用戶「記得做某事」的系統，最終都會因為人的惰性而失敗。** 所以：

- 階段 1 表單是「讓系統能跑起來」的最低門檻
- 真正的價值在階段 2（自動從對話紀錄提取），因為用戶的行為不需要改變 — 他們照常聊天，系統在後台撈
- 階段 3 才是終局：用戶什麼都不用做，系統自動把公開+私人消息整合到一張圖上

建議給 CEO 的溝通框架：「現在先用表單手動輸入累積初始數據，同步開發自動抓取腳本。等腳本就緒，手動輸入只作為補充（特別敏感的消息不走聊天記錄的情況）。」
