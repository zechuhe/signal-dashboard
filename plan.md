## 0. 產品定位

**不是看盤工具，是「消息影響力評估系統」。**

核心邏輯：追蹤「私人消息」→「變成公開消息」→「股價反應」的時間差和影響力，讓用戶在消息剛進來時就能預判影響力等級。

**關鍵洞察**：私人消息的價值 = 時間窗口。任何私人消息最終都會變成公開消息（紙包不住火），但早知道 2 個月 vs 等市場反應完，結果完全不同。

### 0.1 三層架構

| 層 | 定位 | 說明 |
|----|------|------|
| **L1 可視化** | 手段 | 股價圖表 + 事件標記 + 互動展開。讓消息和股價的關聯「看得見」 |
| **L2 數據輸入** | 原料 | 公開訊息 = **系統自動撈**（基礎設施）；私人訊息 = **唯一的手動輸入**（CEO/天使投資） |
| **L3 關聯分析** | 目的 | LLM 分類事件 → 追蹤股價反應 → 建立影響力基準 → 泛化預判 |

**泛化關鍵維度**：公司體量（市值）、業務範圍（單一/多元）、項目占營收比、人員層級、消息類型（人事/財務/政策/供應鏈）

### 0.2 分工鏈條（會議決議）

- **天使投資**：消息來源 + 標的篩選（輸入私人消息）
- **CEO**：數據清洗 + 回測分析 + 微信加密數據導出
- **John**：架構設計 + 工具打造

### 0.3 四階段 Roadmap

| 階段 | 名稱 | 核心交付 | 狀態 |
|------|------|---------|------|
| **P1** | Demo 可視化 | 靜態 Demo HTML + 5 組公開案例 + GitHub | ✅ 完成 |
| **P2** | 手動輸入（過渡） | 私人消息輸入表單 + 股票代碼搜尋（分港/美/A/台） + localStorage | 🔜 下一步 |
| **P3** | 公開訊息自動撈取 | 後端腳本自動抓取財經新聞/公告/港交所/微信公眾號 → 自動產出 events.json | 待排 |
| **P4** | LLM 關聯分析 | 影響力評估模型 + 泛化 + 「私人消息→公開消息」時間差追蹤 | 待排 |

> **P2 是過渡階段**：手動輸入 = 改變用戶行為 = 長期不可持續。但在 P3 自動撈取就緒前，需要手動輸入來累積初始數據和驗證系統。P3 就緒後，手動輸入僅作為補充（特別敏感不走聊天記錄的消息）。

### 0.4 用戶反饋（2026-03-30，CEO + 天使投資首次看 Demo）

1. 需要股票代碼搜尋欄，分港股/美股/A股
2. 需要消息輸入表單（結構化欄位：選擇題/日曆，不能亂輸入）
3. 後台應自動提取關鍵資訊（時間、描述、分類）
4. 長期需後端自動撈取用戶對話紀錄 → 提取 → 自動上傳（John 補充）

**用戶沒提到但一定會遇到的缺口**：
- 消息時效性（3 天後才輸入，時間點已不準）
- 可信度校準（缺乏參考基準）
- 回溯驗證（輸入後想知道「準不準」）
- 消息去重（同一事件多管道聽到）
- 歷史數據冷啟動（大量歷史聊天記錄無法一條條手填）

## 5.1 統一股價數據格式（對齊實際 API）

調研 Yahoo Finance / Alpha Vantage / TWSE / Fugle / Tushare / AKShare / Futu OpenAPI / Tiger 後，統一格式如下：

```json
{
  "meta": {
    "symbol": "HKEX:0001",
    "name": "CK Hutchison",
    "exchange": "HKEX",
    "currency": "HKD",
    "timezone": "Asia/Hong_Kong",
    "price_precision": 2
  },
  "data": [
    {
      "date": "2018-03-16",
      "open": 97.50,
      "high": 98.20,
      "low": 93.10,
      "close": 93.85,
      "volume": 28450000,
      "turnover": 2710000000,
      "adj_close": 93.85,
      "prev_close": 97.30,
      "change": -3.45,
      "change_pct": -3.55
    }
  ]
}
```

**設計決策（對齊主流 API 的交集）**：
| 決策 | 理由 |
|------|------|
| `open/high/low/close/volume` 欄位名 | Yahoo/Fugle/Tiger 共用，最通用 |
| Symbol 格式 `{exchange}:{code}` | 避免歧義（如 `0001` 在 HK/TW 不同意義），Futu 用 `HK.00700` 類似概念 |
| 日期 ISO 8601 `YYYY-MM-DD` | Fugle/AKShare 原生格式，Yahoo 用 Unix timestamp 需轉換，TWSE 用民國年需轉換，Tushare 用 YYYYMMDD 需加 hyphen |
| 值一律 number | TWSE/Alpha Vantage 回傳 string，需轉換 |
| volume 以「股」為單位 | Tushare 以「手（100 股）」計，需 ×100 |
| turnover 以原始貨幣為單位 | Tushare 以「千」計需 ×1000 |
| currency 顯式標記 | 多市場必要，Yahoo meta.currency 有此欄位 |

**交易所代碼對照（擴充用）**：
| exchange | 市場 | 代碼範例 | Yahoo 後綴 |
|----------|------|---------|-----------|
| HKEX | 港股 | 0001, 0700 | .HK |
| NYSE / NASDAQ | 美股 | AAPL, MSFT | 無後綴 |
| TWSE / TPEx | 台股 | 2330, 0050 | .TW / .TWO |
| SSE | A 股上海 | 600519 | .SS |
| SZSE | A 股深圳 | 000001 | .SZ |

## 9. P2 規劃 — 手動輸入 + 股票搜尋（過渡階段）

### 9.1 股票代碼搜尋

- **輸入方式**：選市場（港股/美股/A股/台股）→ 輸入代碼或公司名 → 自動匹配
- **代碼清單**：預載各市場主要股票的 `{code: name}` 對照 JSON（前端 autocomplete）
- **股價拉取**：選定股票後，後端（或前端 proxy）從 Yahoo Finance 拉取歷史日線 → 存入 data/

### 9.2 私人消息輸入表單

| 欄位 | 輸入方式 | 必填 | 備註 |
|------|---------|------|------|
| 關聯股票 | 下拉選擇（已載入的股票） | 是 | 限選已有數據的，避免亂填 |
| 事件標題 | 文字輸入（限 30 字） | 是 | |
| 事件描述 | 多行文字 | 是 | |
| 消息取得日期 | 日曆選擇器 | 是 | 預設今天 |
| 推測事件發生日期 | 日曆選擇器 + 「無法推測」checkbox | 否 | |
| 預期影響方向 | 單選（利好/利空/不確定） | 是 | |
| 信心度 | 滑桿 0-100% | 是 | 預設 50% |
| 消息來源層級 | 單選（董事會/高管/中層/業界/網路） | 否 | |

**存儲**：localStorage + JSON 匯出按鈕。CEO 定期匯出上傳 GitHub 即可更新。

### 9.3 P2 不做的

- 不做公開訊息輸入（那是 P3 自動撈的事）
- 不做後端（純前端 + 靜態 JSON）
- 不做即時股價（仍用 yfinance 預拉取）

## 10. P3 規劃 — 公開訊息自動撈取

### 10.1 架構

```
[多源抓取] → [Python 解析] → [LLM 事件提取+分類] → events.json → [推送 GitHub / 雲端]
                                                                          ↓
[Dashboard 前端] ←── fetch ←── GitHub Pages / Cloudflare Pages
```

### 10.2 輸入源

| 來源 | 自動化可行性 | 負責人 | 備註 |
|------|-------------|--------|------|
| 財經新聞 API（新浪/東方財富） | 高 | John/CEO | 有公開 API |
| 港交所公告（HKEX News RSS） | 高 | John | 結構化數據 |
| SEC/EDGAR（美股） | 高 | John | API 成熟 |
| TWSE 公告（台股 OpenData） | 高 | John | 有公開 API |
| 巨潮資訊（A 股） | 中 | CEO | 需爬蟲 |
| 微信公眾號 | 中 | CEO | 需 RSS 代理或爬蟲 |
| 微信聊天記錄 | 中 | CEO | 加密 DB，CEO 負責導出腳本 |

### 10.3 LLM 事件提取

非結構化文本 → 提取：
- 涉及公司/股票代碼
- 事件類型（人事/財務/政策/供應鏈/法律）
- 事件發生時間
- 預期影響方向

建議 LLM 提取後人工確認再入庫（半自動），避免錯誤數據污染分析模型。

### 10.4 「私人→公開」自動匹配

P3 的核心差異化功能：當系統自動撈到一條公開消息，與已有的私人消息匹配（同公司+同主題），自動計算「時間窗口」（私人消息日 → 公開日）和「股價反應」（公開日前後的價格變動）。

## 11. P4 規劃 — LLM 關聯分析

### 11.1 影響力評估模型

輸入維度：
- 公司體量（市值）
- 業務範圍（單一/多元）
- 事件涉及業務占總營收比
- 人員層級（創始人 > C-Suite > VP > 中層）
- 消息類型（人事/財務/政策/供應鏈）
- 歷史同類事件的影響力分佈

輸出：影響力等級（高/中/低）+ 預期股價反應範圍 + 信心度

### 11.2 泛化目標

不針對特定公司，而是建立跨公司的影響力基準：
- 「市值 50 億的公司，核心項目負責人離職」→ 歷史平均跌 3-5%
- 「市值 5000 億的公司，某部門總監離職」→ 歷史幾乎無波動
- 「油價因中東衝突漲 15%」→ 依業務成本結構，看產業
